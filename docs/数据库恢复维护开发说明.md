# 数据库恢复维护开发说明

## 1. 目标

本次开发完成“数据库恢复 / 维护模块”，并与之前的“安全管理模块”分开实现。

依据需求文档，本模块实现：

- 数据库逻辑备份
- 数据库逻辑恢复
- 数据库日志文件维护

本次不实现：

- 崩溃恢复
- 自动 redo / undo
- 物理文件镜像备份
- 定时备份

## 2. 设计思路

### 2.1 为什么采用逻辑备份

需求文档明确指出：

- 不能只做简单的物理文件拷贝
- 需要实现逻辑备份，或基于日志的恢复

结合当前项目结构：

- 元数据文件是二进制结构体
- 记录文件是文本行
- 解析器已经支持执行 SQL

所以本次采用：

- 逻辑备份：导出为可回放 SQL 脚本
- 逻辑恢复：读取备份脚本并逐条重新执行

这是当前代码结构下实现成本最低、可解释性最强、最适合课程项目答辩的方案。

## 5. 模块职责

### 5.1 BackupManager

`BackupManager` 负责：

- 备份指定数据库到 SQL 脚本文件
- 从 SQL 脚本恢复数据库
- 维护数据库日志文件
- 提供日志文件路径和数据库目录路径

主要接口：

```cpp
bool backupDatabase(const std::string& dbName, const std::string& filePath);
bool restoreDatabase(const std::string& dbName, const std::string& filePath);
bool appendLog(const std::string& dbName, const std::string& sql);
std::string getLogPath(const std::string& dbName) const;
```

## 6. 备份文件设计

### 6.1 备份格式

备份文件是纯文本 SQL 脚本。

示例：

```sql
-- logical backup generated by database_412
-- source database: recover_src
CREATE DATABASE recover_src;
USE recover_src;

CREATE TABLE t1;
ALTER TABLE t1 ADD id INT NOT NULL;
ALTER TABLE t1 ADD name VARCHAR(20);
INSERT INTO t1 VALUES ('1', 'Tom');
INSERT INTO t1 VALUES ('2', 'Jerry');
```

### 6.2 备份内容

当前备份内容包括：

- `CREATE DATABASE`
- `USE`
- `CREATE TABLE`
- `ALTER TABLE ADD`
- `INSERT INTO`

即：

- 数据库结构
- 表结构
- 字段定义
- 全部记录数据

### 6.3 不包含的内容

当前备份不包含：

- `users.db`
- `privileges.db`
- 事务状态
- 运行时内存状态

说明：

- 本模块的恢复对象是“业务数据库”
- 安全模块系统文件目前未纳入逻辑备份范围

## 7. 恢复流程设计

恢复流程如下：

1. 检查目标数据库是否已经存在  
2. 检查备份文件是否存在  
3. 如果磁盘上存在同名但未登记的遗留目录，则先清理  
4. 逐行读取备份文件  
5. 跳过空行和注释行  
6. 如果备份脚本中的库名与目标库名不同，则重写：
   - `CREATE DATABASE ...`
   - `USE ...`
7. 调用 `SQLParser::execute()` 逐条执行  
8. 任何一条失败则停止恢复  

## 8. 日志文件设计

### 8.1 日志文件位置

每个数据库目录下新增一个日志文件：

- `data/<db>/<db>.log`

例如：

- `data/recover_src/recover_src.log`

### 8.2 日志记录内容

当前日志记录成功执行的以下语句：

- `CREATE TABLE`
- `ALTER TABLE`
- `DROP TABLE`
- `INSERT INTO`
- `UPDATE`
- `DELETE FROM`

### 8.3 日志写入时机

日志追加在 `SQLParser::execute()` 成功返回后执行。

这样可以保证：

- 失败语句不会写入日志
- 恢复逻辑读取的脚本语句与真实执行结果一致

补充说明：

- 当前会记录成功执行的 `CREATE TABLE / ALTER TABLE / DROP TABLE / INSERT / UPDATE / DELETE`
- 事务中的 `INSERT / UPDATE / DELETE` 在事务未提交前不会写入 `.log`
- `BACKUP` / `RESTORE` 自身不会写入业务数据库日志

### 8.4 建库时日志文件创建

在 `DatabaseManager::createDB()` 中，数据库目录创建后会立即创建：

- `<db>.log`

这样数据库创建后立刻具备日志落盘能力。

## 9. SQL 语法

### 9.1 逻辑备份

```sql
BACKUP DATABASE recover_src TO 'C:\path\recover_src.sql';
```

格式：

```sql
BACKUP DATABASE <db_name> TO <file_path>;
```

### 9.2 逻辑恢复

```sql
RESTORE DATABASE recover_dst FROM 'C:\path\recover_src.sql';
```

格式：

```sql
RESTORE DATABASE <db_name> FROM <file_path>;
```

说明：

- 当前要求目标数据库不存在
- 如果已存在，会直接失败

## 10. SQLParser 接入

本次在 `SQLParser.cpp` 中新增：

- `BACKUP`
- `RESTORE`

解析流程：

- `BACKUP DATABASE <db> TO <file>` -> `BackupManager::backupDatabase`
- `RESTORE DATABASE <db> FROM <file>` -> `BackupManager::restoreDatabase`

同时保留之前的：

- 用户/权限管理
- DDL / DML
- 事务命令

## 11. execute 返回值收紧

为了让恢复逻辑可靠，本次把 `SQLParser` 内部各命令处理改成“尽量真实返回 bool 成功状态”。

这样带来的好处：

- 备份恢复时能知道某条语句是否真正执行成功
- 失败时可在恢复中断
- 日志只会记录成功执行的语句

## 14. 联调验证

建议最小联调流程如下：

```sql
LOGIN admin 'admin123';
CREATE DATABASE recover_src;
USE recover_src;
CREATE TABLE t1;
ALTER TABLE t1 ADD id INT NOT NULL;
ALTER TABLE t1 ADD name VARCHAR(20);
INSERT INTO t1 VALUES (1, 'Tom');
INSERT INTO t1 VALUES (2, 'Jerry');
BACKUP DATABASE recover_src TO '...\\recover_demo.sql';
RESTORE DATABASE recover_dst FROM '...\\recover_demo.sql';
USE recover_dst;
SELECT * FROM t1;
```

联调结果：

- 备份成功生成 SQL 脚本
- 恢复成功创建新库 `recover_dst`
- 恢复后查询结果与源库一致
- `recover_dst` 中能正确查到两条记录

另外建议补测两类边界数据：

- 含空字符串或未填列的记录，确认恢复后列不移位
- 含单引号或反斜杠的字符串，确认备份脚本可正常回放

## 15. 当前限制

### 15.1 只支持逻辑备份 / 恢复

当前没有实现：

- 页级恢复
- 崩溃后自动恢复
- 日志回放恢复

### 15.2 只支持单条 SQL 输入

和当前项目原有机制一致：

- 一次发送一条 SQL
- 不支持一行里写多条语句再自动拆分执行

### 15.3 备份脚本不包含安全系统文件

当前不会备份：

- `users.db`
- `privileges.db`

### 15.4 恢复不做事务式回滚

如果恢复中途失败：

- 已经成功执行的前半部分不会自动撤销

这点是当前版本的已知限制。

### 15.5 备份范围仍是业务数据

当前备份和恢复只覆盖：

- 数据库元数据
- 表结构
- 记录数据

当前不覆盖：

- 用户和权限系统文件
- 客户端连接状态
- 事务运行时状态

## 16. 后续可扩展方向

- 将 `users.db` / `privileges.db` 纳入系统级备份
- 增加 `BACKUP ALL` 或系统级备份命令
- 支持恢复前自动清空同名数据库
- 基于 `.log` 做 redo / undo
- 支持批量执行脚本
- 支持定时备份

## 17. 答辩时可直接说明的话

可以直接这样概括：

1. 本模块没有采用简单物理拷贝，而是实现了逻辑备份。  
2. 备份文件本质上是一份可回放 SQL 脚本，包含建库、建表、字段定义和插入数据。  
3. 恢复时逐条读取脚本并重新执行，因此恢复结果可验证、可解释。  
4. 同时系统对成功执行的 DDL / DML 维护了 `<db>.log` 日志文件，为后续扩展崩溃恢复预留了基础。
