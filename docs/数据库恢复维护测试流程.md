# 数据库恢复维护测试流程

## 1. 目的

本文档用于从零开始验证当前项目的“数据库恢复 / 维护”模块。

测试目标：

- 验证 `BACKUP DATABASE`
- 验证 `RESTORE DATABASE`
- 验证恢复后结构与数据一致
- 验证数据库日志文件生成与追加

## 2. 测试前准备

### 2.1 关闭后端程序

确保 `database_412.exe` 没有在运行。

### 2.2 准备干净环境

如需从头验证，建议清理：

- `system.db`
- `users.db`
- `privileges.db`
- `data/`

说明：

- 下次启动时会重新初始化默认管理员 `admin / admin123`
- 如果不想删掉原安全管理数据，可只清理测试数据库目录

### 2.3 编译后端

- 解决方案：`database_412.sln`
- 工程：`database_412`
- 配置：`Debug | x64`

生成输出：

- `x64/Debug/database_412.exe`

## 3. 启动系统

### 3.1 启动服务端

运行：

- `x64/Debug/database_412.exe`

预期：

- 服务端成功启动
- 控制台提示监听 `8080`
- 首次启动时提示默认管理员初始化

### 3.2 启动客户端

运行 Qt 客户端工程 `DBMS_Client`，连接服务端后在 SQL 输入框逐条执行。

## 4. 建立源数据库

依次执行：

```sql
LOGIN admin 'admin123';
CREATE DATABASE recover_src;
USE recover_src;
CREATE TABLE t1;
ALTER TABLE t1 ADD id INT NOT NULL;
ALTER TABLE t1 ADD name VARCHAR(20);
INSERT INTO t1 VALUES (1, 'Tom');
INSERT INTO t1 VALUES (2, 'Jerry');
SELECT * FROM t1;
```

预期：

- 所有语句成功
- `SELECT` 可看到两条记录

## 5. 执行逻辑备份

执行：

```sql
BACKUP DATABASE recover_src TO 'C:\Users\asus\Desktop\files\Coding\C_Plus_Plus\visual studio\database_412\docs\recover_demo.sql';
```

预期：

- 备份成功
- 指定路径生成 SQL 脚本

检查备份文件应至少包含：

- `CREATE DATABASE recover_src;`
- `USE recover_src;`
- `CREATE TABLE t1;`
- `ALTER TABLE t1 ADD ...`
- `INSERT INTO t1 VALUES ...`

## 6. 执行逻辑恢复

执行：

```sql
RESTORE DATABASE recover_dst FROM 'C:\Users\asus\Desktop\files\Coding\C_Plus_Plus\visual studio\database_412\docs\recover_demo.sql';
USE recover_dst;
SELECT * FROM t1;
```

预期：

- 恢复成功创建新库 `recover_dst`
- 查询结果与 `recover_src` 一致
- 能看到两条记录

## 7. 验证日志文件

检查以下文件是否存在：

- `data/recover_src/recover_src.log`
- `data/recover_dst/recover_dst.log`

预期：

- 建库后会创建 `.log`
- 源库日志中可看到成功执行的 DDL/DML
- 恢复库日志中也会记录恢复过程中成功执行的 DDL/DML

## 8. 异常场景验证

### 8.1 恢复到已存在数据库

执行：

```sql
RESTORE DATABASE recover_src FROM 'C:\Users\asus\Desktop\files\Coding\C_Plus_Plus\visual studio\database_412\docs\recover_demo.sql';
```

预期：

- 失败
- 提示目标数据库已存在

### 8.2 备份文件不存在

执行：

```sql
RESTORE DATABASE recover_x FROM 'C:\not_exists.sql';
```

预期：

- 失败
- 提示备份文件不存在

## 9. 边界数据建议验证

### 9.1 含空值记录

执行：

```sql
LOGIN admin 'admin123';
USE recover_src;
INSERT INTO t1 VALUES (3, '');
BACKUP DATABASE recover_src TO 'C:\Users\asus\Desktop\files\Coding\C_Plus_Plus\visual studio\database_412\docs\recover_demo.sql';
RESTORE DATABASE recover_empty FROM 'C:\Users\asus\Desktop\files\Coding\C_Plus_Plus\visual studio\database_412\docs\recover_demo.sql';
USE recover_empty;
SELECT * FROM t1;
```

预期：

- 空字符串可被正确恢复
- 列不会错位

### 9.2 含单引号和反斜杠

执行：

```sql
USE recover_src;
INSERT INTO t1 VALUES (4, 'Tom\'s');
INSERT INTO t1 VALUES (5, 'C:\\tmp');
BACKUP DATABASE recover_src TO 'C:\Users\asus\Desktop\files\Coding\C_Plus_Plus\visual studio\database_412\docs\recover_demo.sql';
RESTORE DATABASE recover_escape FROM 'C:\Users\asus\Desktop\files\Coding\C_Plus_Plus\visual studio\database_412\docs\recover_demo.sql';
USE recover_escape;
SELECT * FROM t1;
```

预期：

- 恢复成功
- 含引号和反斜杠的字符串值保持正确

## 10. 验收结论

如果下面这些都满足，可认为数据库恢复维护模块通过基本测试：

1. 能成功导出逻辑备份 SQL 脚本
2. 能从备份脚本恢复出新数据库
3. 恢复后表结构和记录数据与源库一致
4. 数据库目录下能生成 `.log` 文件
5. 成功执行的 DDL / DML 能写入日志
6. 对已存在目标库和不存在备份文件能正确报错
