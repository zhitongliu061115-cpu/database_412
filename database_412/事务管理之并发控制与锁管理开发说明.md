1. 目标
本次实现并发控制与锁管理模块，为 DBMS 提供多事务并发执行时的数据一致性保障。

依据需求文档，本模块实现：

表级锁

行级锁

共享锁（读锁）与排他锁（写锁）

事务结束时自动释放锁

锁等待机制

锁状态查看

2. 本次实现范围
已实现功能：

表级排他锁：INSERT/UPDATE/DELETE 在事务中自动获取表锁

行级排他锁：UPDATE/DELETE 在事务中逐行获取行锁

表级共享锁：SELECT 在事务中获取表级共享锁

锁类型区分：共享锁（读）与排他锁（写）

锁兼容性：共享锁与共享锁兼容，其他组合互斥

锁超时：等待超过 5 秒自动放弃

SHOW LOCKS 命令：查看当前所有锁状态

5. 模块设计
5.1 LockManager 职责
LockManager 负责：

管理所有资源的锁状态

处理锁请求（获取/释放）

维护锁等待队列

检查锁兼容性

提供锁状态查询

5.2 锁类型定义
共享锁（SHARED）：读锁，多个事务可同时持有
排他锁（EXCLUSIVE）：写锁，只能一个事务持有

5.3 锁兼容性矩阵
当前锁为无锁时，请求共享锁允许，请求排他锁允许。
当前锁为共享锁时，请求共享锁允许，请求排他锁等待。
当前锁为排他锁时，请求共享锁等待，请求排他锁等待。

5.4 锁粒度设计
表级锁资源标识格式：table:表名，例如 table:users
行级锁资源标识格式：table:表名:row:行号，例如 table:users:row:0

5.5 锁获取时机
SELECT 在事务中获取表级共享锁，非事务不加锁。
INSERT 在事务中获取表级排他锁，非事务不加锁。
UPDATE 在事务中获取表级排他锁加行级排他锁，非事务不加锁。
DELETE 在事务中获取表级排他锁加行级排他锁，非事务不加锁。

5.6 锁释放时机
COMMIT 释放当前事务持有的所有锁。
ROLLBACK 释放当前事务持有的所有锁。
锁超时等待超过 5 秒自动放弃。

6. 线程安全设计
TransactionManager 中的 currentTxId_ 使用 thread_local 修饰，每个线程拥有独立的事务ID，多客户端并发时互不干扰。

8. 锁行为说明
8.1 共享锁（读锁）
用于 SELECT 操作。多个事务可以同时持有同一资源的共享锁。持有共享锁时，其他事务的写操作会被阻塞。

8.2 排他锁（写锁）
用于 INSERT/UPDATE/DELETE 操作。同一资源只能被一个事务持有排他锁。持有排他锁时，其他事务的读写操作都会被阻塞。

8.3 锁等待
当请求的锁被其他事务持有时，请求事务进入等待队列，每 100ms 检查一次锁是否可用，超过 5 秒超时返回失败。

8.4 事务结束释放
COMMIT 后自动释放所有锁，ROLLBACK 后自动释放所有锁，被等待队列中的事务自动唤醒。

9. 与事务模块的集成
锁管理器和事务管理器协同工作。事务中操作自动获取锁，COMMIT 或 ROLLBACK 时自动释放所有锁。

示例：
BEGIN;
UPDATE users SET age = 26 WHERE id = 1;
COMMIT;

12. 当前限制
无死锁检测：如果两个事务互相等待，会永久阻塞（需手动超时或外部干预）

无锁升级：不支持从共享锁升级为排他锁

无意向锁：表级锁和行级锁独立管理，没有优化

无 MVCC：读操作会阻塞写操作

超时固定：当前固定 5 秒，不可配置

13. 后续可扩展方向
死锁检测（构建等待图，自动回滚牺牲事务）

可配置锁超时时间

意向锁（优化表级+行级混合场景）

多版本并发控制（MVCC，实现读不阻塞写）

手动锁命令（LOCK TABLE / UNLOCK TABLE）

锁等待超时自定义（SET LOCK_TIMEOUT = 3000）
